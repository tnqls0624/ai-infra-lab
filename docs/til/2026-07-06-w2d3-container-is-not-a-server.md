# [W2 D3] 컨테이너는 서버가 아니다 — journalctl 실패로 배운 것

오늘은 코드 대신 리눅스 기본기를 만졌다. `docker run -it ubuntu`로 컨테이너 하나를 띄우고 그 안에서 시스템 상태 확인 명령을 하나씩 돌려봤다.

먼저 자원 확인. `lscpu`로 코어 수를 보니 11개였다. `top`으로 load average를 봤는데, 처음엔 숫자만 읽었지 의미를 몰랐다. 코어가 11개면 1분/5분/15분 load average 각각 최댓값이 11까지 올라갈 수 있다는 뜻이다. 코어 수 대비 비율로 봐야 부하 여부를 판단할 수 있다는 걸 여기서 알았다. `free -m`으로 메모리, `df -h`로 디스크 용량도 같이 확인했다.

여기서 막혔다. `top`을 켜려는데 명령이 없었다. 최소 이미지라 top·htop 같은 도구조차 안 깔려 있어서 `apt update && apt install`로 직접 설치해야 했다. 컨테이너는 기본이 root 계정 + 최소 이미지라는 걸 몸으로 확인한 셈이다.

그다음 `journalctl`을 쓰려다 실패했다. 왜 안 되는지 찾아봤다: plain ubuntu 컨테이너에는 systemd(init)가 없고, 컨테이너의 PID 1은 내가 띄운 셸이다. 로그는 systemd-journald가 관리하는데 그 프로세스 자체가 없으니 당연히 안 된다. 오늘 가장 중요한 발견이다 — 컨테이너는 격리된 프로세스 하나일 뿐, 완전한 서버가 아니다.

권한 실습도 했다.

```
root@1e001715902c:/# touch /tmp/a; chmod 400 /tmp/a; ls -l /tmp/a
-r-------- 1 root root 0 Jul  6 14:23 /tmp/a
root@1e001715902c:/# chmod 600 /tmp/a
root@1e001715902c:/# ls -l /tmp/a
-rw------- 1 root root 0 Jul  6 14:23 /tmp/a
```

400은 소유자 읽기만, 600은 읽기+쓰기 — 숫자 하나가 권한 비트에 바로 매핑된다는 걸 눈으로 봤다. `whoami; id`, `vmstat 1`도 같이 돌려봤다.

오늘 새로 안 것 3개로 정리하면 이렇다.

1) 최소 이미지엔 top조차 없다 — 서버 도구는 당연히 있는 게 아니라 apt로 직접 설치하는 것.
2) journalctl 실패는 systemd 부재 + PID 1이 셸이라는 컨테이너 구조 자체의 문제다.
3) chmod 400/600처럼 숫자가 바로 읽기/쓰기 권한 비트로 대응한다.
